在 ASP.Net MVC 中使用 ELMAH 套件紀錄網站未預期的例外

發表於 2022-10-09 更新於 2022-10-11

ELMAH 是什麼? 為甚麼要使用 ELMAH

系統開發的過程中，免不了會發生一些預期之外的 Exception。若沒有做良好的錯誤處理或記錄，這些發生過的 Exception 必定隨著時間而船過水無痕。若是可以重現的錯誤是還好，重現錯誤後修補即可。怕的是偶發性的 Exception，又或者 User 沒辦法清楚描述出出現 Exception 的情境，這些都是在實務過程中常常會發生的情況。為了要避免這樣的情況，今天介紹一個非常簡單好用的老牌套件 ELMAH。ELMAH 幾乎可以在不用修改任何 Source Code 的情況下，記錄下所有發生的錯誤。並有內建的 Dashboard 可以顯示發生過的所有錯誤，且內容豐富，連 StackTrace 的紀錄都有，讓我們可以有依據紀錄找到錯誤資訊，輕鬆地將 Bug 束手就擒。

ELMAH 的安裝

而 ELMAH 的安裝也很簡單，透過 Nuget 安裝以下套件

  • elmah
  • elmah.corelibrary
  • Elmah.Mvc

安裝完畢後會跳出一個 readme.txt 檔案，簡單說明了 ELAMH 的使用方式，也提及了一些使用上的安全性問題。

同時專案的 Web.config 也會自動產生一些 ELMAH 所需的設定，這些設定內容在 ELMAH 的 GitHub 都可以找到詳細的說明。

ELMAH 的使用

安裝完畢後測試看看 ELMAH 是否已啟用。首先將專案 Run 起來。

Run 起來後在 Domain 後加上 ELMAH.axd 即可進入 Dashboard。
此時 Dashboard 的錯誤清單是空的，因為我們尚未製造出任何錯誤。

在 Url 打上一個不存在的路徑製造一個找不到資源的 Exception。

此時再回到 ELMAH.axd，可以看到 Dashboard 出現了剛才的 Exception。

點擊 Details 後可以看到更詳細的錯誤資訊。

錯誤訊息的儲存方式

這些錯誤訊息預設是儲存在記憶體的(in-memory)，因此只要我們重新啟動專案，可以看到剛才的錯誤紀錄消失了。

而這不是我們所樂見的，我們不會希望這些錯誤紀錄在我們重啟專案，或是重啟 IIS 又或是重新開機後紀錄都消失吧! 因此 ELMAH 除了記憶體的存儲方式也提供了其它方法，像是寫入 XML 檔案或是直接將紀錄寫到 DB 等等。我個人是喜歡將錯誤存到 DB 的方式，因為就能直接在資料庫連線工具中下 SQL 語法查詢出我要找的資訊，較為簡單直覺。而 ELMAH 也支援了大部分主流的資料庫。

將錯誤訊息存入 SQL Server

最後一個範例要說明的是如何將錯誤紀錄直接存入 SQL Server，首先在回到 Nuget 下載另一個套件 elmah.sqlserver。

安裝完成後可以在檔案總管中找到兩個新加入的檔案。其中 Elmah.SqlServer.txt 的文件中寫道連線 SQL Server 的兩個步驟:

  1. 在欲儲存錯誤紀錄的資料庫執行 Elmah.SqlServer.sql
  2. 到 Web.config 中編輯 Connection string

在欲儲存錯誤紀錄的資料庫執行 Elmah.SqlServer.sql。

執行完畢後資料庫會產生一個 Table ELMAH_Error 還有三支 Stored Procedure。

回到 Web.config，安裝完套件 elmah.sqlserver 後會在 Web.Config 中新增兩行程式碼。

這時候只要將剛才的連線資訊寫入 elmah-sqlserver 的 connectionString，錯誤紀錄就會自動寫入資料庫了。

Get & Set 存取子的基本使用

Get & Set 存取子是用來設定 Class 的 Property，根據該屬性存取設定會有以下三種設定方式。

  1. 唯讀: 用 Get 取值，將要回傳的值 return 回去
  1. 唯寫: 用 Set 寫入他的值，寫入的值會存在 value
  1. 可以讀跟寫: 同時寫上 Get & Set

Get & Set 存取子的簡化版本

看到這裡您可能會想也太麻煩了吧!定義一個屬性除了 Public 屬性之外，還需要多定義一組對應的 Private 屬性。所以在 C# 3.0 增加了**自動實作屬性（automatically implemented properties)**，讓我們可以用更簡單的語法，就能設定出能讀能寫的 property。

若是要設定屬性唯讀，在 set 存取子前加上 private 就可以。

Get & Set 存取子變化範例

若是我們要在 Car Class 中建立一個汽油存量的屬性 Gasoline，其屬性值不能低於 0。我們可以再 Gasoline 屬性的 set 存取子進行判斷，如果欲存入的 value < 0，則會直接存入 0，避免油量為負數的狀況。

物件導向(五) 封裝性 Public Private Protected

發表於 2020-02-07 更新於 2022-10-11

封裝性 (Encapsulation) 是物件導向三大特性之一。若是要理解封裝性，可以想像當我們在駕駛汽車時想加速的話，我們不需要理解引擎如何運作將動力傳輸到輪胎上，只需要踩下油門踏板就好，因為加速的知識已經被封裝起來了。再舉一個例子我們想拍照的時候不需要去理解鏡頭是如何捕捉影像在儲存至手機裡，只需要按快門就好，因為按快門至儲存影像這中間的運作，也已經被封裝起來了，透過這兩個例子應該會對封裝性有初步的理解吧!

存取修飾詞

而要善用物件導向的封裝特性，我們首先需要認識一些存取修飾詞，下面將簡單介紹幾個常見的存取修飾詞。

  • Public: 被設定的 class 或其成員可以被任何人所存取。
  • Private: 被設定的 class 或成員，只能在自己的類別結構中被存取。
  • Protected: 被設定的 class 或成員，只可由相同類別或繼承自該類別的程式碼進行存取。

若想知道更多的存取修飾字資訊，可以參考MSDN 存取修飾詞 。

簡介範例的 code example

我們試著把第一段簡介中油門踏板範例透過程式實例化出來，首先會有一個類別 Car，記錄著車輛的各種功能，其中 public 的功能是可以直接被外部所呼叫的，而 private 的功能則只能被 Car 類別內的其他功能所呼叫。

再建立一個駕駛者的類別 Driver，其中有一個方法是踩下油門，而踩下油門後會做的動作就是去呼叫 Car 類別的 public method 加速()。

如此一來 Driver 想加速時只要踩下油門就會加速，而 Car 類別中其他 private method Driver 都不需要了解(也不能呼叫)，因為全部都被封裝在加速() method 了。

無論是程式新手或是有一些開發經驗的工程師，一定都會在範例程式或是專案中常常看到 static 關鍵字。本篇文章將簡單介紹 static 常見的使用情境，還有在使用時需要注意的事項。

靜態方法

靜態方法常出現在專案中的共用方法，像是 helper class 或是 type extention，最方便的地方在於靜態方法不需要被實體化，就可以被使用。下面的範例將示範傳統非靜態類方法與靜態方法被呼叫的方式:

output: 10

呼叫靜態方法，不須要將類別實體化為物件，就可以直接透過類別呼叫。

output: 30

靜態變數

除了方法外，變數也可以在宣告時加入 static 修飾字，一樣是不需要實體化類別就可以被使用，但要特別注意的是，所有該類別實體化的物件，都可以改變靜態變數，而且這個靜態變數將被所有物件共用，下面將示範一個靜態變數的使用情境。

在 Dad 做了存錢，Mom 做了花錢的動作後，因為 accountMoney 為靜態變數，所以會被所有物件 (Dad、Mon) 共用。也因 accountMoney 為靜態變數，所以在輸出 accountMoney 時，我們要直接透過類別 (Family) 來呼叫。

靜態類別

若我們可以確定該類別內的所有成員 (property、method…) 皆為靜態類型，也可以直接將 class 設計為靜態類別。好處在於可以確保類別不會被實體化，且類別內容若出現非靜態成員，編輯器也會在編譯前就先報錯提醒。要設計一個靜態類別，需要注意以下事項:

  • 只能包含靜態類別成員
  • 不允許被實體化為物件
  • 不能被其他類別所繼承

而靜態類別的寫法也相當簡單，在 class 前加上 static 關鍵字即可。

物件導向(三) 建構子及其多載設計

發表於 2020-01-12 更新於 2022-10-11

C# 依照記憶體存取方式的不同，將型別大致分為實質型別 (value type) 與參考型別 (reference type)。也因為這兩種型別在執行存取時運作的模式不一樣，觀念不夠清晰的情況下，常會發生執行結果與預期不相符的情況。這篇文章將簡單說明 value type 與 reference type 在執行記憶體存取的運作情形。

記憶體位置 stack & heap

在說明兩種型別存取過程前，需要先簡單了解記憶體有兩個重要區塊:

  • stack: 用於靜態記憶體配置，必須在編譯時期為已知的資料。
  • heap: 用於動態記憶體配置，使用這裡的記憶體主要是用在編譯時期還不知道大小或個數的資料。

這部分非本章節重點，只要先大致知道記憶體有這兩個不同區塊即可。

C# 中絕大部分的型別都屬於 value type，像是 int、bool、double、decimel…，在記憶體中存取的方式比較單純，下面將用一個簡單的例子來拆解 value type 在記憶體中存取的逐步運作。

要學習物件導向的第一步，就是要先釐清**物件(Object)與類別(Class)**的關係。
以下整理了幾個網路上比喻物件類別之間關係的例子:

  1. 類別是建築物的設計圖，而物件則是依照設計圖設計出來的建築物。
  2. 類別是一本武林秘笈，而物件則是精通武林秘笈的武林高手。
  3. 物件是實體化後的類別。

而我自己的理解是類別是虛擬的，需要將類別實體化(new出來)變成物件，才能被使用，下面將使用一段簡單的程式碼來示範何謂類別，何謂物件。

程式碼說明:

  1. 首先建立一個 Car 類別。
  2. Car 類別中包含三個屬性( brand、color、speed )及一個方法( Run )。
  3. 在主程式 Main 中實體化一個 Car 類別的物件名為 toyota。
  4. 賦予 toyota 這個物件的屬性值。
  5. 執行 Run() 方法，並將結果輸出。

以上就是物件與類別的基本介紹，可以將重點歸納為下面幾個項目:

  • 物件是實體化後的類別。
  • 類別中包含屬性( property )與方法( method )。
  • 要產生物件必須要將類別 new 出來。

看到這裡您可能會想，那我就直接執行 MaMaCook() 就好啦，何須多此一舉宣告一個委派來執行。但呼應文章開頭所說的「委派就是將方法當成參數傳進另一個方法中」，在設計類別時，可能會碰到某個方法需要依照不同情境有不同的處理，但又不想把該處理用 if 或 switch 來處理，因為有可能變化很多，又或是無法預先得知處理的規則，這時我們就可以將方法指派給 howToCook，再將 howToCook 當作參數傳進其他 function。

例如下方範例的 GetFood 方法，其中可能有數十種，甚至數百種的烹煮食物邏輯，我們若把所有邏輯寫在同個 function 中，這個 GetFood 方法勢必會變得又臭又長，不好維護。所以我們將烹煮食物邏輯抽出，包裝在委派 HowToCook 中，並當成參數傳進 GetFood 方法，如此一來， GetFood 方法就只要專注於自己的輸出功能上，不用去在意烹煮食物邏輯。
