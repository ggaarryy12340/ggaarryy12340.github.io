Asp.Net Core 使用 Cookie-based Authentication 進行授權及權限控管

發表於 2023-09-05 更新於 2023-09-07

在 .Net Core 的時代，主流的授權驗證方式有以下三種。

  • Cookie-based Authentication
  • JWT (JSON Web Token)
  • ASP.NET Core Identity

其中實作最簡便的我認為是 Cookie-based Authentication，這種機制類似於以前 .Net Framework 時代的「表單驗證」(Forms Authentication)。僅需加入一些內建的語法，即可完成授權及權限控管的基本功能。而 Cookie-based Authentication 也足以應付中小型網站大部分的需求，只要做好 Cookie 屬性設定後安全性也無須擔心。本篇將記錄下 Cookie-based Authentication 的實作流程。

Startup.cs 設定

先至 Startup.cs 的 ConfigureServices() 加入以下設定:

再至 Startup.cs 的 Configure() 加入以下設定(要注意順序)

  • app.UseCookiePolicy();
  • app.UseAuthentication();
  • app.UseAuthorization();

登入 API

接著到負責處理權限驗證的 Controller 新增 Login API。要做的動作如下:

  1. 到資料庫驗證輸入的資訊是否有吻合的對象
  2. 若有則新增 Claim List 將登入資訊放入
  3. 到權限資料庫領取該使用者的權限並加入 Claim List
  4. 呼叫 Cookie-Based Authentication 登入方法 HttpContext.SignInAsync()

登出設定

登出也很簡單，只要呼叫 HttpContext.SignOutAsync() 即可。

身分驗證

登入登出 API 完成了，接下來就是設定那些功能必須要登入才能讀取，需套用 Authorize 屬性，有以下幾種套用方式

  • 套用在 Class 層級(所有 Class 的服務都要登入才可使用)
  • 套用在 Action 層級(這個 Action 必須登入才可以使用)
  • 套用 Authorize 屬性外還加上 Roles(除了登入還需要有設定的權限才可使用)
  • 匿名驗證: 可以直接套用整個 Class 再針對一些無需登入就可使用的功能(如登入頁面、最新消息) 加上 [AllowAnonymous]

功能實測

在登入前呼叫任一套用 [Authorize] 的 API 會被導到 Startup.cs 設定的 NotLogin API。

呼叫 Login API 候登入成功後，可以看到 Response 多了一組加密的 AspNetCore 的 Cookie。

登入成功後再次呼叫有套用 [Authorize] 的 API，可以成功取得資料了。

呼叫登出 API 後，可以發現 AspNetCore 的 Cookie 消失了。

重新呼叫套用 [Authorize] 的 API 又被導到 NotLogin 了。

其他安全性設定

雖然 Cookie-based Authentication 很方便，但 Cookie 畢竟也會存放在 Client 端，若被有心人士竊取後，將 Cookie 加進自己的 Request 就可以進入我們的網站並取得權限，這聽起來也有點可怕。所以還是建議要在 Cookie 套用 Secure、HttpOnly、SameSite 等屬性，讓這套授權驗證機制能夠更安全。

  • Secure: 限使用 Https
  • HttpOnly: 無法被 JavaScript 調用
  • SameSite: 同網域才能使用

結語

授權驗證幾乎是每個系統必定會應用到的功能，.Net 有內建替我們省下了大量的開發時間，讓我們可以更專注在業務邏輯的開發。試想若是自己造輪子除了會花費大把的時間外，還要擔心在安全性的部分設想的周不周全。今天完成了 Cookie-based Authentication 的實作，預計下一篇會寫開頭提到的 JWT 驗證，也是很主流的驗證方式，不能不知道。

三種生命週期介紹

  • Transient(暫時性): 每次注入都生成新的實體
  • Scoped(範圍姓): 每次 request 產生一個新的實體
  • Singleton(單一性): 程式運行中只會產生一個實體

第一次 Request 執行結果

  • TransientCount: 因為每次注入都產生新的實體，所以只會有 TestController.cs 第 32 行的 +1。
  • ScopedCount: 因為每次 request 只會產生一個實體，所以除了 TestController.cs 第 33 行的 +1，還有 _testDIService.Execute() 內的 +1。
  • SingletonCount: 因為程式運行中只會有一個實體，所以除了有 TestController.cs 第 33 行的 +1，還有 _testDIService.Execute() 內的 +1。

第二次 Request 執行結果

TransientCount 及 ScopedCount 的結果都跟第一次呼叫一樣。但 SingletonCount 變成了 4。這是因為 Singleton 生命週期在程式運行中的只會產生一個實體。所以第一次及第二次 request 都是共用同個實體，第二次 request 前 SingletonCount 已經是 2，之後又跑了兩次 AddOne()，最終結果才會是 4。

結語

以往在 .Net Framework 的時代，要使用 DI 還需另外安裝像是 AutoFac 等等的套件。而在 Core 的時代，微軟直接就將 DI 服務內建了。這代表了 DI 是現今開發的趨勢，降低耦合性，提高可測試性就是 DI 帶來的好處。今天實際撰寫範例程式詳細了解了 DI 的三種生命週期，對此有了更深刻的印象。以後也更能在各種服務間，找到最適合的註冊方式。

Json Patch 介紹

在設計 API 時，更新資料常以 Put 或 Patch 兩種型式進行更新。通常更新完整資料會使用 Put，而更新部分資料則會使用 Patch。而 Json Patch 就是當我們欲更新部分資料時，呈現更新資訊的一種 Json 格式，放在 Body 內在透過 Patch 的方式傳到後端。而後端在接收 Json Patch 格式後進行解析，並依照解析結果進行變更動作。

一組 Json Patch 可以有組更新資訊，在 Json 內以逗點分隔即可。

對稱式加密及非對稱式加密

加密的方法一般分成對稱式加密與非對稱式加密。對稱式加密顧名思義就是分別在提供端及接收端存放一組相同的 Key 供加解密使用。而非對稱加密則是需先在接收端產生公鑰及私鑰後，將公鑰給資料提供端進行加密，而資料接收端在收到加密的資料後再以私鑰解密。單從文字可能不太好理解，可以參考下面 RSA 的加解密流程圖。而非對稱式加密雖然安全性較高，但加解密速度較慢，對稱式加密則反之。沒有好與壞，一切都看需要使用加解密的情境。

在 ASP.Net MVC 中使用 ELMAH 套件紀錄網站未預期的例外

發表於 2022-10-09 更新於 2022-10-11

ELMAH 是什麼? 為甚麼要使用 ELMAH

系統開發的過程中，免不了會發生一些預期之外的 Exception。若沒有做良好的錯誤處理或記錄，這些發生過的 Exception 必定隨著時間而船過水無痕。若是可以重現的錯誤是還好，重現錯誤後修補即可。怕的是偶發性的 Exception，又或者 User 沒辦法清楚描述出出現 Exception 的情境，這些都是在實務過程中常常會發生的情況。為了要避免這樣的情況，今天介紹一個非常簡單好用的老牌套件 ELMAH。ELMAH 幾乎可以在不用修改任何 Source Code 的情況下，記錄下所有發生的錯誤。並有內建的 Dashboard 可以顯示發生過的所有錯誤，且內容豐富，連 StackTrace 的紀錄都有，讓我們可以有依據紀錄找到錯誤資訊，輕鬆地將 Bug 束手就擒。

ELMAH 的安裝

而 ELMAH 的安裝也很簡單，透過 Nuget 安裝以下套件

  • elmah
  • elmah.corelibrary
  • Elmah.Mvc

安裝完畢後會跳出一個 readme.txt 檔案，簡單說明了 ELAMH 的使用方式，也提及了一些使用上的安全性問題。

同時專案的 Web.config 也會自動產生一些 ELMAH 所需的設定，這些設定內容在 ELMAH 的 GitHub 都可以找到詳細的說明。

錯誤訊息的儲存方式

這些錯誤訊息預設是儲存在記憶體的(in-memory)，因此只要我們重新啟動專案，可以看到剛才的錯誤紀錄消失了。

而這不是我們所樂見的，我們不會希望這些錯誤紀錄在我們重啟專案，或是重啟 IIS 又或是重新開機後紀錄都消失吧! 因此 ELMAH 除了記憶體的存儲方式也提供了其它方法，像是寫入 XML 檔案或是直接將紀錄寫到 DB 等等。我個人是喜歡將錯誤存到 DB 的方式，因為就能直接在資料庫連線工具中下 SQL 語法查詢出我要找的資訊，較為簡單直覺。而 ELMAH 也支援了大部分主流的資料庫。

Get & Set 存取子的基本使用

Get & Set 存取子是用來設定 Class 的 Property，根據該屬性存取設定會有以下三種設定方式。

  1. 唯讀: 用 Get 取值，將要回傳的值 return 回去
  1. 唯寫: 用 Set 寫入他的值，寫入的值會存在 value
  1. 可以讀跟寫: 同時寫上 Get & Set

Get & Set 存取子的簡化版本

看到這裡您可能會想也太麻煩了吧!定義一個屬性除了 Public 屬性之外，還需要多定義一組對應的 Private 屬性。所以在 C# 3.0 增加了**自動實作屬性（automatically implemented properties)**，讓我們可以用更簡單的語法，就能設定出能讀能寫的 property。

若是要設定屬性唯讀，在 set 存取子前加上 private 就可以。

Get & Set 存取子變化範例

若是我們要在 Car Class 中建立一個汽油存量的屬性 Gasoline，其屬性值不能低於 0。我們可以再 Gasoline 屬性的 set 存取子進行判斷，如果欲存入的 value < 0，則會直接存入 0，避免油量為負數的狀況。

測試1: 當 Gasoline 存入的值為 100 時。

透過 get set 存取器，就能輕鬆在屬性端設定存取限制。防呆避免不是預期的值被存入屬性中。

物件導向(五) 封裝性 Public Private Protected

發表於 2020-02-07 更新於 2022-10-11

封裝性 (Encapsulation) 是物件導向三大特性之一。若是要理解封裝性，可以想像當我們在駕駛汽車時想加速的話，我們不需要理解引擎如何運作將動力傳輸到輪胎上，只需要踩下油門踏板就好，因為加速的知識已經被封裝起來了。再舉一個例子我們想拍照的時候不需要去理解鏡頭是如何捕捉影像在儲存至手機裡，只需要按快門就好，因為按快門至儲存影像這中間的運作，也已經被封裝起來了，透過這兩個例子應該會對封裝性有初步的理解吧!

存取修飾詞

而要善用物件導向的封裝特性，我們首先需要認識一些存取修飾詞，下面將簡單介紹幾個常見的存取修飾詞。

  • Public: 被設定的 class 或其成員可以被任何人所存取。
  • Private: 被設定的 class 或成員，只能在自己的類別結構中被存取。
  • Protected: 被設定的 class 或成員，只可由相同類別或繼承自該類別的程式碼進行存取。

若想知道更多的存取修飾字資訊，可以參考MSDN 存取修飾詞 。

靜態變數

除了方法外，變數也可以在宣告時加入 static 修飾字，一樣是不需要實體化類別就可以被使用，但要特別注意的是，所有該類別實體化的物件，都可以改變靜態變數，而且這個靜態變數將被所有物件共用，下面將示範一個靜態變數的使用情境。

靜態類別

若我們可以確定該類別內的所有成員 (property、method…) 皆為靜態類型，也可以直接將 class 設計為靜態類別。好處在於可以確保類別不會被實體化，且類別內容若出現非靜態成員，編輯器也會在編譯前就先報錯提醒。要設計一個靜態類別，需要注意以下事項:

  • 只能包含靜態類別成員
  • 不允許被實體化為物件
  • 不能被其他類別所繼承

而靜態類別的寫法也相當簡單，在 class 前加上 static 關鍵字即可。
